# pdbfe Cold-Start Bootstrap

Step-by-step procedure for provisioning a new pdbfe deployment from scratch.
For upgrading an existing deployment, see [Existing DB Upgrades](#existing-db-upgrades).

---

## Prerequisites

- `wrangler` CLI installed and authenticated (`wrangler login`)
- Python 3.12+ with venv (`python3 -m venv scripts/.venv && scripts/.venv/bin/pip install -r scripts/requirements.txt`)
- `sqlite3` available on PATH
- A PeeringDB API key (read-only is sufficient) — set as `PEERINGDB_API_KEY` in `.env`

---

## Step 1 — Provision Cloudflare Resources

Run once per account. Record the IDs printed by each command.

```bash
# D1 — primary PeeringDB mirror database
wrangler d1 create peeringdb

# D1 — authenticated user database (OAuth sessions, preferences)
wrangler d1 create pdbfe-users

# KV — session token store
wrangler kv namespace create SESSIONS

# Vectorize — graph-structural embedding index (1024-dim node2vec, cosine)
wrangler vectorize create pdbfe-vectors --dimensions=1024 --metric=cosine

# Enable metadata filtering on entity type (required for graph search)
wrangler vectorize create-metadata-index pdbfe-vectors \
    --property-name=entity --type=string

# Queue — async task bus between sync and async workers
wrangler queues create pdbfe-tasks

# R2 — logo mirror bucket
wrangler r2 bucket create pdbfe-logos
```

---

## Step 2 — Configure Credentials

Copy `.env.deploy.example` to `.env.deploy` and fill in the IDs from Step 1:

```
CLOUDFLARE_ACCOUNT_ID=<from wrangler whoami>
D1_DATABASE_ID=<peeringdb database_id>
D1_USERDB_ID=<pdbfe-users database_id>
KV_SESSIONS_ID=<SESSIONS namespace_id>
API_DOMAIN=pdbfe.dev          # your custom domain
VECTORIZE_INDEX_NAME=pdbfe-vectors
PDBFE_VERSION=<current version from bump_version.sh>
```

Copy `.env.example` to `.env` and set:

```
PEERINGDB_API_KEY=<your api key>
```

Both files are gitignored. Never commit them.

---

## Step 3 — Generate Wrangler Configs

```bash
./scripts/deploy.sh --generate-configs
```

Reads `.env.deploy` and expands all `wrangler-*.toml.example` files into their
corresponding `wrangler-*.toml` counterparts. Verify the output confirms the
correct `domain` before proceeding.

---

## Step 4 — Bootstrap D1

Downloads public JSON dumps from `public.peeringdb.com`, builds a clean SQLite
database, verifies referential integrity, and imports into D1.

```bash
./scripts/migrate-to-d1.sh --fetch --remote
```

`schema.sql` (generated by the pipeline) already includes all current columns
(`__vector_embedded`, `__logo_migrated`, etc.), so no separate migration step
is required on a cold start.

> If you already have fresh JSON dumps in `database/`, omit `--fetch`.

---

## Step 5 — Backfill Private POC Records

The public JSON dumps only contain `visible=Public` POC records. This step
backfills the non-public records via the authenticated API.

```bash
source .env
python3 scripts/backfill_poc.py | \
    npx wrangler d1 execute peeringdb --remote --yes --file /dev/stdin
```

---

## Step 6 — Deploy Workers

```bash
./scripts/deploy.sh --remote
```

Deploys all 8 workers in dependency order (async consumer before sync producer).
The sync worker will start emitting tasks to `pdbfe-tasks` on its next cron tick
(every 15 min). The async worker will consume them.

---

## Step 7 — Compute Graph Embeddings

Embeds all entities using the node2vec graph pipeline. Run **without**
`--incremental` on a cold start to cover the full dataset.

```bash
source .env
python3 scripts/compute-graph-embeddings.py
```

This may take 30–60 minutes depending on dataset size. After it completes,
the async worker handles ongoing re-embedding via the task queue automatically.

---

## Ongoing Operations

| Task | Command |
|---|---|
| Upgrade existing DB schema | `./scripts/deploy.sh --remote --apply-migrations` |
| Re-sync from scratch (data only) | `./scripts/migrate-to-d1.sh --fetch --remote` |
| Re-backfill private POC records | `python3 scripts/backfill_poc.py \| wrangler d1 execute ...` |
| Purge contaminated vectors | `python3 scripts/cleanup-junk-vectors.py` |
| Re-embed incrementally | `python3 scripts/compute-graph-embeddings.py --incremental` |

---

## Existing DB Upgrades

To apply pending `database/migrations/*.sql` to a live deployment without
a full re-bootstrap:

```bash
./scripts/deploy.sh --remote --apply-migrations
```

The deploy script tracks applied migrations in a `_migrations` D1 table and
skips any already applied. Safe to run repeatedly.
